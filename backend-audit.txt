PURPOSE
-------
A repeatable, end-to-end procedure to find WHY API endpoints (starting with the
companies + jobs feeds) are slow, prove the root cause with data (EXPLAIN, query
log, timings), fix it (indexes / query refactors), and verify the improvement.

Do NOT guess. Every finding in this audit must be backed by one of:
  - an EXPLAIN plan showing a full scan / filesort / temp table, OR
  - a measured query time (ms) from the query log / slow log, OR
  - a measured TTFB from the browser Network tab.

================================================================================
PHASE 0 — SETUP & GROUND RULES (do this first)
================================================================================

0.1  Pick a realistic dataset
     - Audit against PROD-like row counts, not an empty dev DB. Many problems
       (filesort, N+1, correlated subqueries) only show up at scale.
     - Record current row counts so findings are reproducible:
         SELECT
           (SELECT COUNT(*) FROM firm_profiles)  AS firms,
           (SELECT COUNT(*) FROM jobs)            AS jobs,
           (SELECT COUNT(*) FROM users)           AS users,
           (SELECT COUNT(*) FROM applications)    AS applications,
           (SELECT COUNT(*) FROM conversations)   AS conversations,
           (SELECT COUNT(*) FROM messages)        AS messages;
     - If dev data is tiny, seed/clone a snapshot of prod (anonymised) before auditing.

0.2  Enable measurement tooling (choose what's available)
     a) Laravel query log (quickest, no install) — wrap a suspect call:
          DB::enableQueryLog();
          ... run the controller action / hit the endpoint ...
          dd(DB::getQueryLog());   // shows SQL + bindings + time(ms) per query
     b) MySQL slow query log (catches everything, including N+1):
          SET GLOBAL slow_query_log = 'ON';
          SET GLOBAL long_query_time = 0.05;          -- log anything > 50ms
          SET GLOBAL log_queries_not_using_indexes = 'ON';
          SHOW VARIABLES LIKE 'slow_query_log_file';  -- find the file, then tail it
       (Reset long_query_time back when done.)
     c) Laravel Debugbar or Telescope (dev only) — per-request query count + time,
        surfaces N+1 instantly. Install only in a non-prod env.
     d) Browser DevTools -> Network -> click the request:
          - "Waiting (TTFB)" large  => backend/DB  (this audit)
          - "Content Download" large => payload too big / serialization
          - JS main-thread after     => frontend, out of scope here

0.3  Define the pass/fail bar (targets)
     - p50 endpoint TTFB  < 150 ms
     - p95 endpoint TTFB  < 400 ms
     - Queries per request: no endpoint > ~10 queries; ZERO N+1 loops
     - No EXPLAIN row with type=ALL on a table > a few hundred rows
     - No "Using temporary; Using filesort" on a paginated list endpoint

0.4  Freeze a benchmark BEFORE changing anything
     - Capture baseline timings for each target endpoint (Phase 5 method) so the
       "after" numbers are comparable. Save them in the FINDINGS LOG below.

================================================================================
PHASE 2 — INDEX AUDIT (per table)
================================================================================

2.1  Dump existing indexes for every table in the hot path:
         SHOW INDEX FROM firm_profiles;
         SHOW INDEX FROM jobs;
         SHOW INDEX FROM users;
         SHOW INDEX FROM firm_departments;
         SHOW INDEX FROM applications;
         SHOW INDEX FROM conversations;
         SHOW INDEX FROM messages;
     Note any index marked INVISIBLE — the optimizer will NOT use it
     (e.g. jobs.hiring_for is currently INVISIBLE => effectively no index).

2.2  Build the "needed index" list. RULE: a column deserves an index if it is
     used in any of:
        - a JOIN condition          (foreign keys: jobs.firm_id, firm_profiles.user_id, ...)
        - a WHERE equality/range     (verification_status, is_active, status, city, ...)
        - an ORDER BY                (is_premium, created_at, firm_name, ...)
        - a correlated subquery's WHERE
     Composite index column order = (equality columns..., then range/sort column).
     A leading-wildcard LIKE '%x%' CANNOT use an index — note these separately
     (full-text or a search service is the real fix, not a B-tree index).

2.3  Cross-check each target query's columns against 2.1. Every JOIN/WHERE/ORDER
     column with NO usable index is a finding.

2.4  KNOWN GAPS already identified (verify against live DB, then confirm in fixes):
     - firm_profiles: NO index on verification_status, is_premium, created_at,
       city, firm_type, firm_name. (getCompanies filters verification_status,
       sorts is_premium DESC + created_at DESC — all unindexed -> scan + filesort.)
     - jobs: only PRIMARY + INVISIBLE hiring_for. NO index on firm_id, is_active,
       status, created_at. (getJobs joins on firm_id, filters is_active/status.)
     - firm_departments: confirm firm_id is indexed (used in the leftJoin).

================================================================================
PHASE 3 — EXECUTION PLAN ANALYSIS (EXPLAIN every hot query)
================================================================================

3.1  Get the REAL SQL for each target (with bindings filled in). Easiest source:
     the query log from 0.2(a) — copy the exact statement.

3.2  Run EXPLAIN (and EXPLAIN ANALYZE on MySQL 8 for actual timings):
         EXPLAIN <the query>;
         EXPLAIN ANALYZE <the query>;     -- MySQL 8.0.18+, shows real ms + rows
         EXPLAIN FORMAT=JSON <the query>; -- detailed cost model

3.3  How to READ EXPLAIN — red flags to record as findings:
     - type = ALL            -> full table scan (worst). Want ref / eq_ref / range / const.
     - key   = NULL          -> no index used on that table.
     - rows  = large         -> optimizer expects to examine many rows.
     - Extra: "Using filesort"        -> ORDER BY not served by an index.
     - Extra: "Using temporary"       -> GROUP BY/DISTINCT built a temp table.
     - Extra: "Using join buffer"     -> join with no usable index (block nested loop).
     - A DEPENDENT SUBQUERY line       -> correlated subquery re-runs per outer row (N+1 in SQL).
     - Big gap between EXPLAIN's estimated rows and the table's real selectivity
       -> stale stats; run  ANALYZE TABLE <t>;  and re-check.

3.4  Pagination cost: Laravel paginate() runs a SECOND COUNT(*) query. For a
     GROUP BY query the count wraps the whole thing as a subquery and is often the
     SLOWEST query of the request. EXPLAIN that count separately. Consider
     simplePaginate() (no count) or a cached/approximate total for big lists.

3.5  Correlated subqueries in SELECT (e.g. getCompanies' current_openings count):
     these execute once PER returned row. EXPLAIN will show them as DEPENDENT
     SUBQUERY. Flag every one — replace with a single grouped JOIN or a batched
     whereIn lookup for the paginated page only.

================================================================================
PHASE 4 — N+1 & APPLICATION-LEVEL JOIN AUDIT
================================================================================

4.1  Count queries per request. With Debugbar/Telescope, or by counting
     DB::getQueryLog(). A list endpoint that runs (1 + N) queries where N scales
     with rows returned = N+1. Confirm by loading page size 12 vs 50 and watching
     the query count change.

4.2  Common N+1 shapes to hunt in the controllers:
     - foreach ($rows as $r) { DB::table(...)->where('id', $r->x)->first(); }
     - per-row peer/firm/profile lookups while formatting a list
     - per-row file URL / count / status resolution
     Fix pattern: collect the ids, do ONE whereIn() into a keyed map, resolve from
     the map inside the loop (the pattern already used in MessagingController's
     bulk getConversations refactor).

4.3  Over-selecting: `select firm_profiles.*` pulls every column (incl. large TEXT
     like about) even when the card needs 8 fields. Narrow the SELECT for list
     endpoints to cut row width + sort/temp-table cost.

4.4  JSON columns in WHERE/LIKE (e.g. exposure_type LIKE '%dept%'): can't use a
     normal index; note as a separate class. Options: generated column + index,
     a normalized join table, or JSON functions with a functional index.

================================================================================
PHASE 5 — BENCHMARK (measure before & after, same conditions)
================================================================================

5.1  Warm + repeat: hit each endpoint 5-10x, discard the first (cold) call, record
     median TTFB. Keep filters/body identical between before/after runs.
         # example: time the raw endpoint
         curl -s -o /dev/null -w "TTFB=%{time_starttransfer}s TOTAL=%{time_total}s\n" \
              -X POST https://<host>/getCompanies -H "Content-Type: application/json" -d '{}'
     Run the same with a typical filter payload (city + sort) and with search text.

5.2  Also record at the SQL layer: total query count and summed query ms per
     request (from the query log) — that isolates DB time from PHP/serialization.

5.3  Capture results in the FINDINGS LOG so the fix's impact is provable.

================================================================================
PHASE 6 — FIXES (apply, then re-run Phase 3 + 5 to confirm)
================================================================================
Order of attack = highest leverage, lowest risk first:

  (1) Add the missing indexes (see candidate DDL below). Re-EXPLAIN. Cheapest win.
  (2) Replace correlated SELECT subqueries with grouped JOINs / batched whereIn.
  (3) Remove N+1 loops via whereIn maps.
  (4) Narrow SELECT columns on list endpoints.
  (5) Address pagination COUNT cost (simplePaginate / cached totals) if it's hot.
  (6) For leading-wildcard search: FULLTEXT index or a dedicated search path.

CANDIDATE INDEX DDL (verify column names against live schema first; add to
db_changes.txt WITH rollback before running on prod):

  -- firm_profiles: directory filter + sort
  ALTER TABLE firm_profiles
    ADD INDEX idx_fp_verif_premium_created (verification_status, is_premium, created_at),
    ADD INDEX idx_fp_city       (city),
    ADD INDEX idx_fp_firm_name  (firm_name);

  -- jobs: join + filter + sort (kills the per-firm openings subquery scan)
  ALTER TABLE jobs
    ADD INDEX idx_jobs_firm_active            (firm_id, is_active),
    ADD INDEX idx_jobs_active_status_created  (is_active, status, created_at);

  -- firm_departments: leftJoin key (only if missing)
  ALTER TABLE firm_departments
    ADD INDEX idx_fd_firm (firm_id);

  ROLLBACK:
    ALTER TABLE firm_profiles  DROP INDEX idx_fp_verif_premium_created,
                               DROP INDEX idx_fp_city, DROP INDEX idx_fp_firm_name;
    ALTER TABLE jobs           DROP INDEX idx_jobs_firm_active,
                               DROP INDEX idx_jobs_active_status_created;
    ALTER TABLE firm_departments DROP INDEX idx_fd_firm;

NOTES ON INDEXING SAFELY
  - Indexes speed reads but slightly slow writes + use disk; only add ones the
    EXPLAIN proves are used. Re-EXPLAIN after adding to confirm `key` now picks them.
  - On large tables, ADD INDEX locks/copies — run in a maintenance window or use
    pt-online-schema-change / ALGORITHM=INPLACE where supported.
  - After adding, run  ANALYZE TABLE firm_profiles, jobs;  to refresh optimizer stats.

================================================================================
PHASE 7 — ENVIRONMENT FACTORS (rule out before blaming code)
================================================================================
  - DEV: `php artisan serve` is SINGLE-THREADED. With Reverb + Vite + queue + several
    parallel API calls, requests serialize and FEEL slow but are fine under PHP-FPM
    in prod. Re-test under a real FPM/nginx stack before drawing conclusions.
  - Run `php artisan config:cache` (env()/config not cached can add overhead).
  - Confirm DB is local/low-latency to the app server (network RTT per query
    multiplies N+1 pain).
  - Check `BROADCAST_CONNECTION` / Reverb is reachable — a blocked broadcast on a
    WRITE path can stall those endpoints (read feeds shouldn't broadcast, but verify).
  - MySQL: confirm innodb_buffer_pool_size is large enough to hold the hot tables;
    a cold buffer pool turns index reads into disk reads.

Here are my critical findings from inspecting the actual queries (FirmController.php) and the index definitions in your DB dump. Ranked by severity — each is evidence-backed, not speculation.

🔴 Critical
1. jobs table has effectively zero useful indexes
From the dump: jobs has only PRIMARY + a hiring_for index marked INVISIBLE (the optimizer ignores invisible indexes entirely). There is no index on firm_id, is_active, status, or created_at.
→ getJobs joins jobs → firm_profiles on firm_id and filters is_active/status — all full scans. This is the single highest-impact gap.

2. current_openings is a correlated subquery — an N+1 inside SQL
getCompanies does (select count(*) from jobs where jobs.firm_id = firm_profiles.id and is_active=true) in the SELECT. It runs once per firm row, and because jobs.firm_id is unindexed (finding #1), each execution full-scans the entire jobs table. Compounds multiplicatively as both tables grow.

3. firm_profiles filter + sort columns are unindexed
Dump shows firm_profiles indexes only PRIMARY, is_branch, parent_firm_id. getCompanies filters verification_status = 'approved' and orders by is_premium DESC, created_at DESC — none indexed → full scan + filesort on every request. Note: both the verification_status filter and the is_premium DESC sort were added recently — this is very likely why it "started feeling slower."

🟠 High
4. GROUP BY + GROUP_CONCAT forces a temp table + filesort, and paginate(12) adds a second COUNT(*)
The leftJoin firm_departments + GROUP_CONCAT(...) GROUP BY firm_profiles.id materializes a temp table and sorts it. Then paginate() runs a separate COUNT(*) wrapping that grouped query — often the slowest query of the request.

5. getJobs resolves the firm by firm_name (unindexed) on every call
Line 1467: firm_profiles->where('firm_name', $request->company)->value('id') — firm_name has no index, so a scan just to translate name→id. Plus the student path adds a platform_settings lookup and a student_profiles fetch per request.

🟡 Medium
6. Leading-wildcard search can't use any index
Search does LIKE '%term%' across about, services_offered, description, etc. B-tree indexes are useless for leading wildcards — these are always scans. Real fix is FULLTEXT or a search path, not a normal index (only matters when a search term is present).

7. select firm_profiles.* over-fetches
List cards need ~8 fields but the query pulls every column including large TEXT (about), widening rows and inflating the temp-table/filesort cost in #4.

Bottom line: the slowness is real and structural, not a feeling. The dominant causes are missing indexes on jobs.firm_id / firm_profiles.verification_status+is_premium+created_at and the per-row current_openings subquery. Findings #1–#3 alone likely account for most of the regression, and all three are low-risk to fix.
